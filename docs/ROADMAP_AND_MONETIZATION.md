# DevLife 8-bit — Дорожная карта, улучшения и монетизация

Документ объединяет взгляд **игрока**, **лида разработки** и **монетизации** с планом развития приложения.

---

## 1. Взгляд игрока

### Что уже хорошо

- Понятный онбординг: выбор профессии, быстрый старт.
- Единый стиль: пиксельный шрифт, карточки, консистентные отступы.
- Ясный главный экран: статы, три действия (Учиться / Задания / Отдых), последнее событие.
- Задания как одна точка входа: задача, каналы, мини-игра в одном месте.
- Страница «Как играть» и навигация без лишнего шума.
- Мини-игра «Фикс багов» с управлением под тач и повтором после проигрыша.

### Что улучшить

| Область | Проблема | Рекомендация |
|--------|----------|--------------|
| **Онбординг** | Игрок быстро попадает на главный экран, но не понимает, куда жать дальше. «Как играть» — отдельная страница, до неё надо дойти. | Добавить пошаговый онбординг с подсветкой: «Вот твои статы», «Вот действия», «Вот задания», «Вот навыки». Использовать Telegram BackButton и состояния `phase` в zustand. |
| **Обратная связь** | После действия (учёба, задача) не всегда очевидно, что изменилось. | Сохранить/усилить анимацию дельт (EXP, деньги, энергия) на главной и в заданиях; краткий тост/подсветка «+X EXP». |
| **Прогресс до собеседования** | Не видно, сколько осталось до порога EXP. | На главной или в карточке статов: «До собеседования: 23 / 50 EXP» и прогресс-бар. |
| **Собеседование** | Раздел «в разработке» — нет ключевой механики прогресса. | Реализовать MVP: 3 вопроса, расчёт шанса, успех/провал, кулдаун. Без этого карьерная лестница не ощущается. |
| **Энергия** | Неочевидно, когда придёт следующая единица энергии (реген 1 раз в 2 часа). | Показать таймер/подсказку: «+1 энергия через 1ч 20м» или «Энергия восстанавливается в фоне». |
| **Отдых** | Кулдаун 20 мин — игрок может не понять, почему кнопка неактивна. | Уже есть подсказка «Отдых через N мин» — убедиться, что она видна на мобильном и обновляется. |
| **Навыки (древо)** | Текущий список навыков выглядит сухо и «админским» списком; непонятно, какие билды и синергии. | Перерисовать экран навыков как древо: группы, зависимые уровни, подсказки по билдам (например «фронт + перф», «тесты + надёжность»). Добавить эффект от веток в явном виде: «+5% дохода», «+5% к шансу собеса». |
| **Обучение в геймплее** | Сейчас игра даёт настроение, но почти не даёт реальных знаний о профессии. | Вшить микро-обучение: подсказки в онбординге, пояснения к навыкам, события с краткими объяснениями («что такое code review»), вопросы собеседования, которые отражают реальные темы. |
| **Награды** | Подписка на каналы и фикс багов дают EXP, но не видно «сколько заработал за сессию». | Опционально: краткая сводка «За эту сессию: +X EXP, +Y денег» при возврате на главную или в заданиях. |
| **Рейтинг/социальность** | Нет сравнения с другими. | Дальше: лидерборд по уровню/EXP/деньгам (указано в CONCEPT как v0.2). |

### Приоритет для игрока

1. **Собеседование** — без него нет ощущения карьеры.
2. **Прогресс до собеседования** — понятная цель.
3. **Онбординг уровня «проведи за руку»** — чтобы сразу понимать, куда жать.
4. **Улучшение фидбека** по действиям и энергии.
5. **Навыки и обучение** — сделать древо навыков и вплести микро-обучение.

## 2. Взгляд лида разработки

### Сильные стороны

- **Next.js App Router**, React 19, TypeScript — современный стек.
- **Расширяемая система действий**: реестр действий, типы simple/minigame, один API `POST /api/player/action` с payload.
- **Разделение слоёв**: `lib/game` (логика), `lib/db`, API routes, компоненты; хуки `usePlayer`, `useTelegram`.
- **Переиспользуемые компоненты**: `PlayerProfileCard`, `PlayerActionsCard`, pixelact-ui.
- **Supabase** для БД и миграции в коде.
- **Zustand** для глобального состояния — минимально и предсказуемо.

### Техдолг и риски

| Область | Проблема | Рекомендация |
|---------|----------|--------------|
| **API Stars** | `POST /api/stars` — заглушка, монетизация не подключена. | Реализовать приём Telegram Stars, проверку платежа, начисление энергии (см. раздел монетизации). |
| **API Interview** | `POST /api/interview` — заглушка. | Реализовать старт/отправку ответов, расчёт шанса, обновление уровня/репутации, кулдаун. |
| **RLS Supabase** | В миграциях политики «using (true)» или отсутствуют — нет разграничения по пользователю. | Ввести сервисную роль только для API; либо JWT с telegram_id и политики по `auth.uid()` / кастомному claim. |
| **Тесты** | Нет автотестов. | Добавить unit-тесты для `lib/game` (energy, action, interview), интеграцию для критичных API (auth, action, player). |
| **Ошибки и логирование** | Часть ошибок только в console. | Единый формат ответов API (например `{ ok, error?, code? }`), при необходимости — логирование в сервис (Sentry и т.п.). |
| **Константы монетизации** | STARS_* в constants, но не используются в коде. | Подключить в API stars и в UI (кнопки покупки энергии). |
| **Валидация payload** | В action принимается `payload?: unknown` — нет строгой схемы по типу действия. | По шагам: типизировать payload для fix_bugs, partner_subscribe (channelId); при росте — zod/йоду схемы в API. |

### Масштабирование

- Добавление новых действий: запись в реестр + обработчик + при необходимости страница/мини-игра — архитектура уже позволяет.
- Новые профессии/навыки/события — конфиги в `lib/game`, без хардкода в API.
- Рейтинг/лидерборд: отдельная таблица или материализованное представление по `level`, `exp`, `money`; кэш при необходимости.

### План по качеству кода

1. Включить и довести до зелёного состояния линтер/формат (уже есть).
2. Добавить тесты на ключевые формулы (energy, interview chance, income).
3. Реализовать Stars и Interview API, убрать TODO.
4. Ужесточить RLS или явно задокументировать «API только с сервисной ролью, auth через cookie».

## 3. Монетизация

### Текущее состояние

- **Концепция (CONCEPT.md):** Telegram Stars за энергию: +20 энергии → 29 Stars, полное восстановление → 9 Stars.
- **Код:** константы `STARS_ENERGY_BOOST`, `STARS_ENERGY_BOOST_PRICE`, `STARS_FULL_RESTORE_PRICE`; API `POST /api/stars` — заглушка.
- **В игре:** нет экрана/кнопок покупки энергии за Stars.

### Что нужно для подключения Stars

1. **Bot API:** использовать [Telegram Bot API: pay via Stars](https://core.telegram.org/bots/api#stars) — создание инвойса, получение pre_checkout и успешной оплаты (webhook или getUpdates).
2. **Backend:**  
   - При успешной оплате (payload с product_id/типом) — идентифицировать игрока (telegram_id из payment), начислить энергию (до макс. или +N).  
   - Хранить факт платежа (таблица `payments` или лог), чтобы не начислять дважды.
3. **Frontend:**  
   - Кнопка «Купить энергию» (например в карточке статов или в отдельном блоке «Буст»).  
   - Открытие инвойса Telegram (Mini App или ссылка на бота с invoice).  
   - После оплаты — обновить игрока (refresh или WebApp callback).
4. **Цены:** оставить 29 Stars за буст и 9 Stars за полное восстановление как в константах, при необходимости скорректировать после тестов.

### Дополнительные идеи монетизации (после MVP Stars)

| Идея | Описание | Приоритет |
|------|----------|-----------|
| Снятие кулдауна отдыха | За Stars сразу снова «Отдых» без ожидания 20 мин. | Средний |
| Доп. попытка собеседования | За Stars вторая попытка в тот же «день» без потери репутации. | Средний |
| Косметика / темы | Скины интерфейса, другой пиксель-стиль. | Низкий |
| Расширенная подписка | Ежемесячная подписка: +макс. энергия, бонус к регену, эксклюзивные события. | После валидации MVP |
| Реклама партнёров | Уже есть подписка на каналы — можно монетизировать размещение каналов. | Низкий (партнёрская модель) |

### Риски монетизации

- Перекос в pay-to-win: если за Stars можно купить слишком много прогресса, теряется смысл геймплея. Ограничивать бусты по энергии и попыткам собеседования.
- Неочевидность ценности: игрок должен понимать, за что платит («+20 энергии», «полная энергия»). Короткие формулировки на кнопках и в описании.

## 4. План развития по релизам

Ниже — логичные **релизы**, каждый даёт игроку и продукту целостный апдейт. Внутри релизов задачи можно выполнять параллельно, но выпускать лучше пачкой.

### Релиз 1 — Полный базовый цикл (без монетизации)

Цель: игрок может **расти по карьерной лестнице** от действий до собеседования и понимает, что происходит.

| # | Задача | Ответственный слой | Оценка |
|---|--------|---------------------|--------|
| 1.1 | Реализовать собеседование: API (старт, ответы, шанс, уровень, кулдаун), UI (вопросы, результат, кулдаун). | Backend + Frontend | 2–3 дня |
| 1.2 | Показать прогресс до собеседования на главной (EXP до порога, прогресс-бар или текст). | Frontend | 0.5 дня |
| 1.3 | Улучшить фидбек по действиям (дельта EXP/деньги/энергия, таймер регена энергии). | Frontend | 0.5–1 день |
| 1.4 | Unit-тесты на формулы (energy, interview, income). | Dev | 1 день |

**Результат релиза:** ощущается «игровой цикл»: учусь/делаю задачи → коплю EXP → вижу, сколько до собеса → прохожу собеседование, меняется уровень и доход.

### Релиз 2 — Онбординг и навыки как обучающий опыт

Цель: новый игрок **понимает интерфейс** и **чувствует, что чему‑то учится** про профессию.

| # | Задача | Ответственный слой | Оценка |
|---|--------|---------------------|--------|
| 2.1 | Онбординг 2.0: пошаговая подсветка элементов (статы, действия, задания, навыки) при первом входе. | Frontend | 1–2 дня |
| 2.2 | Навыки 2.0: перестроить экран навыков под древо/ветки с явными эффектами (+% дохода, +% к шансу собеседования). | Frontend + Game logic | 2–3 дня |
| 2.3 | Вшить обучение: описания навыков, событий и вопросов собеседования так, чтобы давали реальное знание о профессии. | Контент + Frontend | 2–4 дня |

**Результат релиза:** онбординг ведёт игрока по интерфейсу, навыки выглядят как понятное древо билдов, а тексты и вопросы дают небольшой, но реальный образовательный эффект.

### Релиз 3 — Монетизация v1 (Telegram Stars)

Цель: аккуратно добавить **покупку энергии** за Stars, не ломая баланс.

| # | Задача | Ответственный слой | Оценка |
|---|--------|---------------------|--------|
| 3.1 | Реализовать приём Telegram Stars: инвойс, webhook/обработка успешной оплаты, начисление энергии. | Backend | 1–2 дня |
| 3.2 | Таблица/лог платежей (id, telegram_id, product_type, amount, created_at). | Backend + миграция | 0.5 дня |
| 3.3 | UI: блок «Буст энергии» на главной или в заданиях (кнопки «+20 энергии», «Полное восстановление» с ценой в Stars). | Frontend | 0.5–1 день |
| 3.4 | Документация: как тестировать Stars в тестовом режиме Telegram. | Docs | 0.5 дня |

**Результат релиза:** появляется первый платёжный сценарий — купить энергию, когда хочется поиграть ещё, но без агрессивного pay‑to‑win.

### Релиз 4 — Удержание и социальность

Цель: добавить **поводы возвращаться** и минимальную социальность.

| # | Задача | Ответственный слой | Оценка |
|---|--------|---------------------|--------|
| 4.1 | Рейтинг/лидерборд (топ по уровню или EXP), отдельная страница или виджет. | Backend + Frontend | 1–2 дня |
| 4.2 | Ежедневные/еженедельные цели (опционально): «Выполни 3 задачи», «Пройди собеседование» — бонус EXP/деньги. | Backend + Frontend | 1–2 дня |

**Результат релиза:** игрок видит себя относительно других и имеет короткие цели «на сегодня», что повышает удержание.

### Релиз 5 — Контент и монетизация 2.0

Цель: расширить **контент** и аккуратно добавить **продвинутые платные фичи**.

| # | Задача | Ответственный слой | Оценка |
|---|--------|---------------------|--------|
| 5.1 | Новые действия/мини-игры в реестре (по желанию продукта), в том числе «Решение сложной задачи» по концепту из `docs/CONCEPT.md`. | Все | по задаче |
| 5.2 | Платные бусты: снятие кулдауна отдыха, доп. попытка собеседования за Stars. | Backend + Frontend | 1 день |
| 5.3 | Расширение событий и контента по профессиям (см. CONCEPT). | Контент + Backend | по объёму |

## 5. Итоговые приоритеты

1. **Игрок:** Собеседование → прогресс до собеседования → онбординг → фидбек по действиям и энергии → навыки и обучение.
2. **Лид:** Реализация Interview и Stars API, тесты на формулы, приведение RLS/безопасности в порядок.
3. **Монетизация:** Подключить Telegram Stars (энергия), затем добавить кнопки в UI и зафиксировать план в этом документе.

После выполнения Фазы 1 и Фазы 2 приложение будет иметь полный игровой цикл и первую работающую монетизацию; Фазы 3–4 — рост удержания и выручки.
