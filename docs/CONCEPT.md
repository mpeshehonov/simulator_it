# DevLife 8-bit — Концепция проекта

> Карьерный симулятор в ретро-стиле. Путь от безработного до Lead в IT.

---

## Жанр и стиль

- **Жанр:** текстово-событийный симулятор с прокачкой
- **Тон:** ирония над айти-сферой
- **UI:** pixel-интерфейс (pixelact-ui), тёмный фон, пиксельный шрифт, зелёные/неоновые акценты

---

## Профессии (5)

| ID | Название |
|----|----------|
| frontend | Фронтенд-разработчик |
| backend | Бэкенд-разработчик |
| qa | QA-инженер |
| devops | DevOps-инженер |
| uiux | UI/UX-дизайнер |

У каждой профессии: 3 ветки навыков, свой пул событий, свои формулы бонусов.

## Ресурсы игрока

| Ресурс | Описание |
|--------|----------|
| 💰 Деньги | Доход за задачи |
| ⚡ Энергия | Макс 10, тратится на действия |
| ⭐ Репутация | Влияет на шанс собеседования |
| 📘 EXP | Опыт для повышения уровня |
| 📊 Уровень | Карьерная ступень (0–5) |

---

## Игровой цикл

1. **Учиться** — +EXP, −Энергия  
2. **Выполнить задачу** — +Деньги, +EXP, случайное событие  
3. **Отдохнуть** — +Энергия  
4. **Собеседование** — при достаточном EXP, шанс повышения  

Цикл: **действие → событие → ресурсы → прогресс**.

**MVP:** 10 на профессию + 10 универсальных + 10 для уровней выше Джуниора.

| Тип | Примеры |
|-----|---------|
| Позитивные | «Оптимизировал бандл — +2 EXP +150$» |
| Нейтральные | «Митинг затянулся — -1 Энергия» |
| Негативные | «Прод упал — -2 Репутация», «Клиент недоволен — -100$» |

---

## Механика собеседования

- 3 вопроса игроку  
- **Формула шанса:** Базовый 40% + бонус навыков + бонус репутации + случайность  
- **Успех:** повышение уровня, рост дохода, новые события  
- **Провал:** −Репутация, кулдаун 1 игровой день  

---

## Экономика

- **Доход за задачу:** Базовый × множитель уровня × множитель навыков  
- **EXP за задачу:** 5–15 случайно  
- **Энергия:** 1 действие = 1 энергия, макс 10, реген по времени

## Мини-игры

### 1. «Фикс багов»

- **Цель:** дать быстрый, динамичный геймплей в духе аркады и привязать его к задачам «починить прод».  
- **Механика:** мини‑Space Invaders, где игрок стреляет по «баго‑эмодзи» и набирает очки за фикс.  
- **Интеграция:** вызывается из задания соответствующего типа, по завершении возвращает количество очков, которые конвертируются в EXP/деньги/репутацию по формуле в `lib/game`.

### 2. «Решение сложной задачи»

- **Формат:** поле из плиток (например, 5×5), под которыми спрятаны «правильные шаги» и «ошибки».  
- **Механика риска:** игрок по очереди открывает плитки.  
  - Если открыта **ошибка** — раунд мгновенно проигран, награда за раунд = 0.  
  - Если открыт **правильный шаг** — счёт растёт; игрок может продолжить рисковать или «забрать награду» и закончить раунд.  
- **Сложность:** задаётся размером поля и соотношением safe/trap:  
  - Лёгкая — много правильных шагов, мало ошибок.  
  - Средняя — баланс safe/trap.  
  - Сложная — мало правильных шагов, поле «заминировано».  
- **Гейм‑цикл:**  
  1. Экран старта с описанием задачи и выбором сложности.  
  2. Экран поля с сеткой плиток, счётчиком текущей награды и кнопкой «Забрать награду».  
  3. По клику по плитке — открытие: safe увеличивает счёт, trap завершает раунд с провалом.  
  4. Экран результата: успех (зафиксированная награда) или провал (0 награды), кнопка «Сыграть ещё».  
- **Подход к реализации:**  
  - Отдельный React‑компонент `SolveComplexTaskGame` по тому же паттерну, что и `FixBugsGame` (проп `onComplete(score)` для интеграции в общую логику действий).  
  - Генерация поля при старте: массив ячеек с нужным количеством safe/trap, перемешивание, преобразование в матрицу.  
  - Состояние каждой ячейки (`hidden/revealed`) и обработчик клика, меняющий `score` и состояние игры.  
  - UI на CSS‑grid без canvas: крупные плитки‑кнопки, адаптированные под тач, с `select-none` и запретом зума (как в первой мини‑игре).
